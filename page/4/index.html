ThreadLocal学习

Posted on 2018-08-01 | In Note

对比理解

ThreadLocal与类的普通成员变量

类的普通成员变量是线程共享的，而ThreadLocal是线程私有的。
也就是说对于一个类对象来说，当线程结束的时候ThreadLocal中的数据需要重新设置，而普通成员变量要等到类对象生命周期结束才被释放。

所以ThreadLocal中保存的数据作用域在线程中，类的普通成员变量作用域在类对象生命周期中，类的静态成员变量生命周期在类被加载释放过程中。

ThreadLocal与Synchronized

这两个都是为了解决变量在多线程中访问冲突的问题。但是两者处理方式不同。

  • Synchronized同步机制采用了“以时间换空间”的方式，为该变量加上锁，让不同的线程排队访问；
  • ThreadLocal采用了“以空间换时间”的方式，为每一个线程都提供了一份变量，因此可以同时访问而互不影响。

Synchronized直接为变量加锁，一次只允许一个线程进入临界区读写变量，这样保证了没有读写冲突。
ThreadLocal通过initialValue方法存了一份变量的拷贝到map中，访问变量的时候通过get从map中查询保存的副本即可，但是如果需要设置值的话调用set方法

源码分析

ThreadLocalMap

每一个Thread对象都有一个ThreadLocalMap对象，这个ThreadLocalMap持有对象的弱引用（方便GC回收）。
ThreadLocalMap以当前的threadlocal对象为key，以真正的存储对象为value。get时通过threadlocal实例就可以找到绑定在当前线程上的对象。

ThreadLocalMap中的节点

Entry便是ThreadLocalMap里定义的节点，如果这里使用普通的key-value形式来定义存储结构，实质上就会造成节点的生命周期与线程强绑定，只要线程没有销毁，那么节点在GC分析中一直处于可达状态，没办法被回收。（摘自博客）

ThreadLocalMap的存储结构

ThreadLocal需要维持一个最坏2/3的负载因子。由于ThreadLocalMap使用线性探测法来解决散列冲突，所以实际上Entry节点数组在程序逻辑上是作为一个环形存在的。

使用

为什么要用static ThreadLocal
通过学习链接中的SimpleDateFormat线程安全分析可以看出来，static ThreadLocal能够做到线程独享，比方法独享在并发环境中能减少不少创建对象的开销。

在一个读取数据库数据导出到excel文件的例子当中，每次处理一个时间信息的时候，就需要创建一个SimpleDateFormat实例对象，然后再丢弃这个对象。大量的对象就这样被创建出来，占用大量的内存和jvm空间。所以可以考虑定义static对象，这样就不会重复创建销毁对象。
但是SimpleDateFormat不是线程安全的，在format函数中调用了calendar.setTime(date)来修改calendar状态，在多线程环境下会引发线程安全问题。如果通过synchronized为对象加锁，会导致其他线程阻塞等待，对多线程性能有影响。

综合考虑使用static ThreadLocal，ThreadLocal将SimpleDateFormat的对象变为线程独享（而不用在线程中定义线程变量），static保证不会重复创建销毁对象。

String和StringBuilder

Posted on 2018-07-25 | In Note

可变性

String创建的字符串是不可变的，而StringBuilder通过创建的字符串可变。

String对象内容

String对象s在创建的时候先查看常量池中是否有”hello”，有则指向它，没有就创建一个”hello”，再指向它，但当重新对s赋值为”world”时，常量池中的”hello”并没有改变，java会重新开辟一个内存存储新值”world”，并令s指向”world”；

https://www.javaspecialists.eu/talks/oslo09/ReflectionMadness.pdf

StringBuilder对象内容

在创建StringBuilder对象时，java实际上在堆中创建了字符类型数组char[]，存储完”hello”后通过append方法在增加”world”，并未开辟新的内存区，形参和实参指向同一区域，因此形参对对象内容进行变化，实参也会跟着变化；

StringBuilder实际创建了一个默认16字符长的char型数组(亦可指定长度)：char[] value

若为对象赋值时，所存储的字符串长度未超过所定义的char数组长度，则按顺序存储相应字符，若超过所定义的数组长度，则自动扩充其长度，因此说StringBuffer是可变的；

StringBuilder自动扩容

当为StringBuilder对象增加内容时，会先计算所需最小空间：minimumCapacity=count+len;并与原数组长度的2倍+2即newCapacity进行比较，取较大值并为数组value重新赋值，赋值采用的是数组复制的方式进行；

常见的数组复制有两种方法：一为：Arrays.copyOf()，另一为：System.arraycopy()，差别在于后者可能存在下标越界的问题，其实前者也是通过调用后者的方法来实现的：

String “+” 运算符

编译器在解析String”+”运算时，会转换成StringBuilder，调用append()方法，再通过toString()赋值，因此循环运算时，就不断发生创建对象和产生垃圾，这过程消耗了资源，造成了效率的下降；

如果直接把String的”+”运算写成形如：s=""+1+1+1……，而不是通过循环来不断调用变量，则编译器会直接求出其字面量，运行时不会再转换为StringBuilder

hashCode

String和StringBuilder的hashCode表现不同

通过hashCode函数可以看出来两个内容相同的String对象s和s2的hashCode值相同，而同一个String对象经过赋值之后的hashCode值不同了。

分别介绍他们的hashCode函数的实现：

StringBuilder.hashCode()

StringBuilder没有重载Object类的hashCode()函数，所以对StringBuilder的hashCode分析较为简单

Object.hashCode是一个native方法，看不到源码。

在博客中给出了使用jni来通过获取java对象和指针的内存地址。

String.hashCode()

String的hashCode函数进行了重载

String类的hashCode的计算过程在注释中已有体现：hash = s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]

转一篇博客来解释为什么出现 31 这个常量

结论是：

  1. 31作为一个不大不小的质数能够有效的降低哈希算法的冲突率，同时不会因为指数爆炸导致的整数溢出
  2. 31有一个很好的特性，即乘法运算可以被移位和减法运算取代，来获取更好的性能：31 * i == (i << 5) - i，现代的 Java 虚拟机可以自动的完成这个优化。

hashCode与对象的内存地址

通过Object.hashCode()上面的注释可以看出来，这个方法的典型实现是通过把对象的内部地址转为为整数。但是实际上hashCode是一个辅助作用的数值，而不是实际的内存地址。
hash值是根据一个哈希算法得到的，理论上来说哈希算法会有冲突，所以会存在两个不一样的对象的哈希值相等。

如果要获取Java对象的内存地址，可以有如下方式：

  • 通过上面的介绍我们知道了Object.hashCode()是一个native方法，所以可以通过jni来使用C++获取对象的内存地址
  • 使用sun.misc.Unsafe类来直接操作Java内存空间

Spring AOP学习

Posted on 2018-07-16 | In Note

基本概念

AOP 框架其实可以将它理解为一个拦截器框架，如果它拦截了一个类，那么它就拦截了这个类中所有的方法。在项目中往往只需要拦截特定的方法就行了，为了解决所有方法被拦截这个问题，引入了切面的概念。

| 概念 | 作用 |
| 切点（PointCut） | 在哪个地方进行切入,它可以指定某一个点，也可以指定多个点 |
| 通知（advice） | 在切入点干什么，切面的实际实现，打日志、执行缓存、处理异常等等 |
| 切面（aspect） | PointCut + Advice 形成了切面Aspect要实现的交叉功能，是系统模块化的一个切面或者领域，如日志记录 |
| 代理 | 将通知用到目标对象后创建的对象，应用系统的其他部分不用为支持代理对象而改变 |
| 目标对象（target） | 被通知的对象 |
| 织入 | 将切面应用到目标对象从而创建一个新代理对象的过程，织入发生在目标对象生命周期的多个点上。在配置interceptionForNames时发生 |
| 引入 | 为类添加新方法和属性 |

当通过代理对象去实现aop时，获取的ProxyFacotoryBean是什么类型？
返回的是一个代理对象，如果目标对象实现了接口，则spring使用jdk动态代理技术；如果没有实现接口，则使用CGLib技术。

代理

静态和动态是由代理产生的时间段来决定的。静态代理产生于代码编译阶段，动态代理产生于运行阶段。

我们以一个例子来理解代理：打日志

静态代理

这种代理方式需要编写固定的接口，并且需要实现接口中的每一个方法。

动态代理

我们不想实现接口中的每一个方法，希望能够编写代理作用于所有的函数上。这样可以通过运行期的反射机制实现。

这种做法有一个局限，只有实现了接口的类才能为其实现代理。

CGLib

CGLib在运行时期（非编译时期）生成被代理对象的子类，并重写了被代理对象的所有方法，从而作为代理对象。
相当于CGLib帮我们完成了静态代理。
（注意：final类无法生成子类，所以无法使用CGLib）
