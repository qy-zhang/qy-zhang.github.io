      锁分类

      MySQL有三种锁的级别：页级、表级、行级
  • 表级锁：开销小，加锁快；不会出现死锁；锁定粒度大，发生锁冲突的概率最高,并发度最低。
  • 行级锁：开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低,并发度也最高。
  • 页级锁：开销和加锁时间界于表锁和行锁之间；会出现死锁；锁定粒度界于表锁和行锁之间，并发度一般

mysql的事务支持与存储引擎有关，MyISAM不支持事务，INNODB支持事务
行级锁并不是直接锁记录，而是锁索引，如果一条SQL语句用到了主键索引，mysql会锁住主键索引；如果一条语句操作了非主键索引，mysql会先锁住非主键索引，再锁定主键索引。

死锁判断 wait for graph

参考进程死锁的四个条件

  1. 互斥：一个资源只能被一个进程使用
  2. 请求和保持：一个进程因请求资源而阻塞时，对已获得的资源保持不放
  3. 不可剥夺：不能抢占其他进程的资源
  4. 循环等待：

innodb将各个事务看为一个个节点，资源就是各个事务占用的锁，当事务1需要等待事务2的锁时，就生成一条有向边从1指向2，最后行成一个有向图。

innodb在两个事务相互等待时，当一个等待时间超过设置的某一阀值时，对其中一个事务进行回滚，另一个事务就能继续执行。

死锁产生举例

这个update语句会执行以下步骤：

  1. 由于用到了非主键索引，首先需要获取idx_1上的行级锁
  2. 紧接着根据主键进行更新，所以需要获取主键上的行级锁
  3. 更新完毕后，提交，并释放所有锁。

如果在步骤1和2之间突然插入一条语句：update user_item .....where id=? and user_id=?
这条语句会先锁住主键索引，然后锁住idx_1。此时一条语句获取了idx_1上的锁，等待主键索引上的锁；另一条语句获取了主键上的锁，等待idx_1上的锁，这样就出现了死锁。

《重构 改善既有代码的设计》

Posted on 2018-12-17

重构原则

重构的定义

重构：对软件内部结构的一种调整，目的是在不改变软件可观察行为的前提下，提高其理解性，降低其修改成本。

根据定义可以看出，准备动手修改前需要考虑清楚，修改的结果是 提高了程序可读性？还是降低了程序的修改成本（提升可扩展性）？

目标是：可读、可扩展

一定要与性能优化区分开来，两者目标不一致，采取的手法和遵循的原则会有些不一致。

两顶帽子

使用重构技术开发软件时，会有两种状态：

  • 添加新功能：尽可能不修改既有代码，只管添加功能。针对新功能可能会对测试用例进行调整。
  • 重构：不能添加新功能，不能修改测试用例。

这两个状态会在开发过程中不断切换。

为何重构

  • 代码结构的流失是累积性的

    越难看出代码所代表的设计意图，代码的结构就会腐烂的越快。定期重构首先可以加深开发者对代码的理解，其次可以为代码结构保鲜

    以mermaid服务的货架模板为例，长久未被修改过的代码只会让维护者更加不想去看，代码的设计者也无法保证能够完整的阐述出之前的设计意图。

  • 重构提升编程速度
    良好的设计是快速开发的根本

程序具有两面价值：今天可以为你做什么、明天可以为你做什么。修复错误、添加新特性只是在关注今天的价值，还需着眼明天。

何时重构

  • 三次法则

    事不过三，三则重构

  • 添加功能时重构

    添加功能时为了帮助自己理解需要修改的代码。在开发的过程中不断理清代码的结构

  • 修补错误时重构

    代码不够清晰，不能快速发现bug，

  • code review后重构

间接层与重构

计算机科学是这样一门学科：它相信所有问题都可以通过增加一个间接层来解决

间接层的价值：

  • 允许逻辑共享
  • 分开解释意图和实现
  • 隔离变换
  • 封装条件逻辑：多态消息，降低代码重复度并提高弹性

重构的难题

  • 数据库
  • 修改接口：已发布的接口重构后打上 Deprecated，为整个包定义一个异常基类。

何时不该重构

  • 代码不能正常运作时重写
  • 项目已近最后期限

代码的坏味道

决定何时重构、何时停止和知道重构机制如何运转一样重要。

受到刚出生的女儿的气味影响，Kent Beck提出用味道来形容重构时机。

重构的记录格式

按照标准划分，每个重构手法都有如下五个部分：

  1. 名称：
  2. 简短概要：简单介绍此重构手法的适用场景，以及他所做的事情。这部分可以帮助你更快速的找到所需要的重构手法
  3. 动机：为什么需要这个重构？什么情况下不应该使用这个重构？

  5. 范例：以一个十分简单的例子来说明此重构手法如何运作

  • 使用<property>元素，实际上是让Spring执行一次setter方法

Git合并commit

Posted on 2018-11-15 | In Blog

为什么需要合并commit

在本地开发分支，推荐多使用交互式rebase整理commit。这样别的开发者只会看到你整理之后的结果，保证干净、清洁。

在一次功能开发中多次提交了代码，但是很多commit是重复、无用的，在合并到master之前建议合并一些commit

比如在一次开发过程中想要整理一下代码。

  • 第一次提交 9ba7300 中删掉了一些注释代码
  • 第二次提交 4300cdb 中发现仍然有一些注释代码，接着又修改代码进行了一次提交，commit偷懒写了update
  • 第三次提交 e5a1c8b 中删掉了一些无用的xml配置

在后续的代码开发过程中如果想要找回删除的xml配置，可能需要逐一查看这三条commit记录，来找到有用的信息。
实际上这三条commit可以被合并成一条，这一次commit的内容就是删除一些代码和配置。

如何合并

进入vim界面后主要关注两条命令：

  • pick 会执行这个commit
  • squash 这个commit会被合并到前一个commit

vim配置

vim command mode下通过 p 快捷键可以将commit状态修改为pick（默认状态），通过 s 快捷键可以将commit状态改成 squash
但是在vim command mode中按下 s 键的效果是删除当前字符并进入 Insert mode

在 .vimrc 中配置键绑定

web.xml中servlet url配置

Posted on 2018-10-24 | In Note

四种匹配规则

精确匹配

url与设置的url-pattern完全匹配

比如servletA 的url-pattern为/test，servletB的url-pattern为/*，这个时候，如果我访问的url为 http://localhost/test ，这个时候容器就会先进行精确路径匹配，发现/test正好被servletA精确匹配，那么就去调用servletA，也不会去理会其他的 servlet了。

路径匹配

以/*结束，匹配所有满足路径的 url

先最长路径，再最短路径

servletA的url-pattern为/test/*，而servletB的url-pattern为/test/a/*，此 时访问 http://localhost/test/a 时，容器会选择路径最长的servlet来匹配，也就是这里的servletB。

扩展名匹配

匹配固定后缀名的url

默认匹配

匹配顺序由上至下，url匹配上一个规则就不会往下走

注意

路径匹配与默认匹配

/和/*的区别：
<url-pattern>/</url-pattern> 不会匹配到*.jsp，即：*.jsp不会进入spring的DispatcherServlet类。
<url-pattern>/*</url-pattern> 会匹配*.jsp，会出现返回jsp视图时再次进入spring的DispatcherServlet类，导致找不到对应的controller所以报404错。

从上图可以看出返回ModelAndView会再次进入DispatchServlet，所以当/*匹配到/home这种路径时，返回了/META-INF/view/home.jsp，又命中了扩展名匹配，还是会被DispatchServlet转发，此时就找不到对应的controller了。

回退到git pull刚拉取代码的样子

通常会有想要找到文件原始的样子的需求。在这里分情况介绍Git撤销修改

已修改未add

当修改了文件，还没有通过git add命令将修改内容

此时可以直接通过编辑器的快捷键（一般是ctrl+z或者cmd+z）撤销修改直至文件颜色显示为无变化。
也可以通过命令直接修改

已add未commit

当修改了文件，并通过git add命令将文件添加到stage了
此时需要通过git reset命令撤回修改

git reset会丢弃提交记录，HEAD^代表HEAD的父提交记录，此参数可以替换为任意commit id。

已commit未提交

git reset –hard commit_id

git reset --hard HEAD^
会直接撤销已经提交的全部文件到编辑之前版本，并且不会有任何操作记录，最好不要使用。

git revert commit_id

git revert命令会生成一个新的版本并提交，通过git log可以查看得到，推荐使用。

已提交

一篇网络博客介绍git reset的三个参数使用

soft mixed hard

–soft 、–mixed以及–hard是三个恢复等级。

  • 使用–soft就仅仅将头指针恢复，已经add的缓存以及工作空间的所有东西都不变。
  • 如果使用–mixed，就将头指针恢复掉，已经add的缓存也会丢失掉，工作空间的代码什么的是不变的。
  • 如果使用–hard，那么一切就全都恢复了，头指针add的缓存消失，代码也恢复到以前状态。
