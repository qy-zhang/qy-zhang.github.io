MySQL锁学习

发表于 2019-01-08
  • 锁分类
    • 死锁判断 wait for graph
    • 死锁产生举例
    • 参考文档

      锁分类

      MySQL有三种锁的级别：页级、表级、行级
  • 表级锁：开销小，加锁快；不会出现死锁；锁定粒度大，发生锁冲突的概率最高,并发度最低。
  • 行级锁：开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低,并发度也最高。
  • 页级锁：开销和加锁时间界于表锁和行锁之间；会出现死锁；锁定粒度界于表锁和行锁之间，并发度一般

mysql的事务支持与存储引擎有关，MyISAM不支持事务，INNODB支持事务
行级锁并不是直接锁记录，而是锁索引，如果一条SQL语句用到了主键索引，mysql会锁住主键索引；如果一条语句操作了非主键索引，mysql会先锁住非主键索引，再锁定主键索引。

死锁判断 wait for graph

参考进程死锁的四个条件

  1. 互斥：一个资源只能被一个进程使用
  2. 请求和保持：一个进程因请求资源而阻塞时，对已获得的资源保持不放
  3. 不可剥夺：不能抢占其他进程的资源
  4. 循环等待：

innodb将各个事务看为一个个节点，资源就是各个事务占用的锁，当事务1需要等待事务2的锁时，就生成一条有向边从1指向2，最后行成一个有向图。

innodb在两个事务相互等待时，当一个等待时间超过设置的某一阀值时，对其中一个事务进行回滚，另一个事务就能继续执行。

死锁产生举例

这个update语句会执行以下步骤：

  1. 由于用到了非主键索引，首先需要获取idx_1上的行级锁
  2. 紧接着根据主键进行更新，所以需要获取主键上的行级锁
  3. 更新完毕后，提交，并释放所有锁。

如果在步骤1和2之间突然插入一条语句：update user_item .....where id=? and user_id=?
这条语句会先锁住主键索引，然后锁住idx_1。此时一条语句获取了idx_1上的锁，等待主键索引上的锁；另一条语句获取了主键上的锁，等待idx_1上的锁，这样就出现了死锁。

      回退到git pull刚拉取代码的样子

      通常会有想要找到文件原始的样子的需求。在这里分情况介绍Git撤销修改

已修改未add

当修改了文件，还没有通过git add命令将修改内容

此时可以直接通过编辑器的快捷键（一般是ctrl+z或者cmd+z）撤销修改直至文件颜色显示为无变化。
也可以通过命令直接修改

已add未commit

当修改了文件，并通过git add命令将文件添加到stage了
此时需要通过git reset命令撤回修改

git reset会丢弃提交记录，HEAD^代表HEAD的父提交记录，此参数可以替换为任意commit id。

已commit未提交

git reset –hard commit_id

git reset --hard HEAD^
会直接撤销已经提交的全部文件到编辑之前版本，并且不会有任何操作记录，最好不要使用。

git revert commit_id

git revert命令会生成一个新的版本并提交，通过git log可以查看得到，推荐使用。

已提交

一篇网络博客介绍git reset的三个参数使用

soft mixed hard

–soft 、–mixed以及–hard是三个恢复等级。

  • 使用–soft就仅仅将头指针恢复，已经add的缓存以及工作空间的所有东西都不变。
  • 如果使用–mixed，就将头指针恢复掉，已经add的缓存也会丢失掉，工作空间的代码什么的是不变的。
  • 如果使用–hard，那么一切就全都恢复了，头指针add的缓存消失，代码也恢复到以前状态。
