MySQL查询区分大小写

Posted on 2019-05-21

介绍

我们知道使用 WHERE 来执行匹配是不区分大小写的，同时在使用 LIKE 谓词来进行模糊匹配的时候也没有区分大小写
解决此问题有两种方式：BINARY 和 COLLATE

  1. 在查询的字符串前面加上BINARY运算符
  2. 建表的时候对相应列加上BINARY
  3. 在查询时指定collation
  4. 在建表时指定collation

BINARY

BINARY 不是函数，而是一个类型转换运算符，用来强制将它后面的字符串修改为二进制字符串，这样在进行字符串的比较的时候就会区分大小写了

collate

在MySQL中，对于Column Collate其约定的命名方法如下：
| 命名方式 | 介绍 |
| - | - |
| _bin | 表示的是binary case sensitive collation，也就是说是区分大小写的 |
| _cs | case sensitive collation，区分大小写 |
| *_ci | case insensitive collation，不区分大小写 |

问题描述

背景：一个thrift服务提供的接口比较核心但是排查问题比较困难，希望能够有一个http接口方便查询。
方案：通过json串来mock出thrift请求参数，然后通过json反序列化成一个thrift model，这样就可以实现http接口查询了
但是在为http接口传入json串的时候遇到了400报错

鉴于此学习了一下http的400相关返回状态码。

400错误码一般是请求参数错误，可能是请求头太大，如果是使用的jetty容器，可以设置jetty请求表单内容大小。

http请求中的长度限制

URL长度限制

在Http1.1协议中并没有提出针对URL的长度进行限制
但是在ng和servlet容器中一般都会配置默认的url长度限制

nginx的配置参数为client_header_buffer_size和large_client_header_buffers
jetty的请求配置参数为requestHeaderSize
tomcat的请求配置参数为maxHttpHeaderSize

对于get请求，在url的长度限制范围之内，（GET）请求的参数个数没有限制
Post数据的长度限制与url长度限制类似，也是在Http协议中没有规定长度限制，长度限制可以在服务器端配置

ng对于请求大小的限制

请求行不能超过一个缓冲区（默认8k）的大小，否则会将414(Request-URI Too Large)错误返回给客户端。
请求头字段也不能超过一个缓冲区的大小，或者400(Bad Request)错误返回给客户端
请求体不能超过默认的1M，否则返回413 (Request Entity Too Large))

jetty的限制

jetty对于请求头大小也有限制，默认8k（or 6k），超过限制则会返回413

SQL基础

语法

  • 多条SQL语句必须分号分隔
  • SQL语句不区分大小写，建议对所有SQL关键字使用大写，而对所有列和表名使用小写，这样做会使代码更易于阅读和调试
  • 在处理SQL语句时，其中的所有空格都被忽略。SQL语句可以在一行上给出，也可以分成许多行便于阅读和调试

主键

主键（primary key）一列（或一组列），其值能够唯一区分表中每一行

唯一标识表中每行的这一列（或这组列）成为主键。虽然并不总是都需要主键，但是建议每个表都具备一个主键，以便于以后的数据操纵和管理。

主键值规则

MySQL本身强制规定：

  • 任意两行都不具有相同的主键值
  • 每个行都必须具有一个主键值（主键列不允许NULL值）

主键的最好习惯

  • 不更新主键列中的值
  • 不重用主键列的值
  • 不在主键列中使用可能会更改的值。（例如如果使用供应商名字作为主键以标识某个供应商，当该供应商合并和更改其名字时，必须更改这个主键）

通配符过滤

HTTP 406 not acceptable错误学习

Posted on 2019-03-13

从问题入门

问题描述

想为一个thrift服务添加http接口，实际上这个服务已经通过bootstrap程序启动了jetty容器，jetty端口也起来了，所以只用配置一些spring mvc相关内容就可以了。但是在添加了一个controller后，发现调用接口前端显示 HTTP 406 not acceptable

原因

406一般原因是客户端请求的数据类型和从服务端接收来的数据的类型不一致


从上图可以看出，状态码406
请求头（Request Headers）中的请求信息是json格式
响应头（Response Hraders）中却发现返回信息的格式是“text/html”

请求信息与响应信息不符合，前端无法解析，需将结果转换成json格式返回给前台。

解决

使用fastjson，引入jar包后配置，在spring-servlet.xml中加上配置即可

jackson加载简介

spring mvc配置了<mvc:annotation-driver/>后 ，如果 classpath 里面包含 jackson 包，则自动注册MappingJackson2HttpMessageConverter，从而支持json 输出。

Spring中是在AnnotationDrivenBeanDefinitionParser中解析<mvc:annotation-driver/>

通过源码可以看出，通过判断jackson包的ObjectMapper和JsonGenerator是否存在来确定jackson包被引入，被引入了就添加一个HttpMessageConverter

解析完成之后，在mvc启动时默认会加载下面的几种HttpMessageConverter，相关代码在 org.springframework.web.servlet.config.annotation.WebMvcConfigurationSupport中的addDefaultHttpMessageConverters方法中

如何影响前端展示

http的基础约定

SpringMVC启动时会自动配置一些HttpMessageConverter，接收到http请求时，从这些Converters中选择一个符合条件的来进行Http序列化/反序列化。

  1. 决定resp.body的Content-Type的第一要素是对应的req.headers.Accept属性的值，又叫做MediaType。如果服务端支持这个Accept，那么应该按照这个Accept来确定返回resp.body对应的格式，同时把resp.headers.Content-Type设置成自己支持的符合那个Accept的MediaType。服务端不支持Accept指定的任何MediaType时，应该返回错误406 Not Acceptable.
    例如：req.headers.Accept = text/html，服务端支持的话应该让resp.headers.Content-Type = text/html，并且resp.body按照html格式返回。
    例如：req.headers.Accept = text/asdfg，服务端不支持这种MediaType，应该返回406 Not Acceptable。
  2. 如果Accept指定了多个MediaType，并且服务端也支持多个MediaType，那么Accept应该同时指定各个MediaType的QualityValue，也就是q值，服务端根据q值的大小来决定这几个MediaType类型的优先级，一般是大的优先。q值不指定时，默认视为q=1.
    Chrome的默认请求的Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/webp,image/apng,/;q=0.8，表示服务端在支持的情况下应该优先返回text/html，其次是application/xhtml+xml.
    前面几个都不支持时，服务器可以自行处理 /，返回一种服务器自己支持的格式。
  3. 一个HTTP请求没有指定Accept，默认视为指定 Accept: /；没有指定Content-Type，默认视为 null，就是没有。当然，服务端可以根据自己的需要改变默认值。
  4. Content-Type必须是具体确定的类型，不能包含 *.

RequestResponseBodyProcessor实现了HandlerMethodReturnValueHandler、HandlerMethodAgumentResolver接口，
所以具备了处理参数、处理返回值的能力。RequestResponseBodyProcessor能够解析用@RequestBody注解的参数和通过使用HttpMessageConverter读取并写入请求体或响应来处理用@ResponseBody注解的方法的返回值。

handleReturnValue()方法获取inputMessage,outMessage对象，调用writeWithMessageConverters()方法，让消息转换器对消息进行下一步处理

private方法导致CGLib空指针

Posted on 2019-02-01 | In Note

问题描述

commonapi项目中TestController中访问private方法会抛空指针，public方法就正常

结论

TestController被cglib动态代理，cglib实现是继承被代理类，所以cglib代理生成的对象中不会有private方法

而@RequestMapping是可以将访问的url映射到private方法上的，当访问public方法时，spring容器中的通过cglib动态代理生成的TestController对象中可以继承public方法，所以可以正常访问

当访问private方法时，在cglib动态代理生成的TestController对象中没有这个方法，会使用原生的未被代理的对象，此时这个对象里的属性 redisDealCacheClient 等都没有被spring容器管理，所以是null

印证

实际到代码中确认：

  1. 断点打到 AbstractClassGenerator.generate(AbstractClassGenerator.ClassLoaderData data) 方法，找到字节数组

主要关注 byte[] b = this.strategy.generate(this); 这一句，断点打到这儿

  1. 在服务启动的时候打断点，spring容器在启动的时候需要实例化bean，断点设置当遇到TestController这个bean时停止

  2. 将字节数组转换输出

  3. 然后将encode出来的value值拷贝出来，再通过程序解码

  4. 最后将cc.class放到IntelliJ中即可查看通过Cglib动态代理生成的类对象所包含的内容了。
    里面是没有private方法的

几点疑问

  1. 为什么controller类会被cglib代理
  2. aggregation代码中同样在TestController写private方法，可以正常访问，不报空指针

正常情况下Controller类被Spring容器初始化之后，是不会被cglib动态代理的，所以写private方法一样可以正常访问，不会有空指针问题
一定是commonapi项目中使用了什么组件，使得所有的Controller都被cglib动态代理，而aggregation项目中没有使用到

通过查看commonapi中的调用堆栈，确定是ControllerLogCollector这个切面进行了处理，这个切面是mdp-boot-starter-log组件中的。

为了印证这个想法，在c-aggregation项目中也配置上mdp-boot-starter-log
首先同commonapi一样，在c-aggregation的 spring-servlet.xml 中加上了下面两个bean，然后添加了文件mdplog.properties

查看bean的类HttpAutoConfiguration ，发现在这个bean中实例化了ControllerLogCollector

发现还是没有被cglib代理，将上面两个bean的配置放到父servlet applicationContext.xml 中再次启动项目

问题复现，private方法中注入的对象为null
