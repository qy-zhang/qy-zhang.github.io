单元测试介绍

Posted on 2017-03-22 | In Note

单元测试(unit testing)，是指对软件中最小可测试单元进行检查和验证。是典型的白盒测试。

黑盒测试：已知产品的功能设计规格，可以进行测试证明每个实现了的功能是否符合要求。数据驱动
白盒测试：已知产品的内部工作过程，可以通过测试证明每种内部操作是否符合设计规格要求，所有内部成分是否以经过检查。逻辑驱动

单元测试的目的

单元测试的目的是尽早在尽量小的范围内暴露错误。

测试不能保证一个程序是完全正确的，但是能够使我们尽早的发现程序的bug和不足。bug隐藏的时间越长修复这个bug的代价越大。
代码的作用就是能够重复运行。当软件变得越来越大，为了添加一些新功能或者进行一些小改动时，很容易带来一些问题使得程序无法正常运行。手动调试十分耗时。
在程序编写的过程中，我们可以一边编写程序，一边创建测试单元来自动完成测试的工作。当修改代码后，单元测试自动完成所有的测试功能。
更有甚者比较推崇测试驱动的开发(TDD)，测试驱动的开发尝试首先编写单元测试，然后再完成实际的代码。当项目结束后，单元测试还将是不错的文档。

单元测试的实现

单元测试就是在判断程序运行的值是否和预期的值一致。

单元测试不是简单的测试代码，大多数开发人员都认为自己写了单元测试，实际上只是在“测试代码”。随着项目的不断演进，“单元测试”能够一直存在并发挥价值，而“测试代码”只会被一行一行注释掉，直到某个接手的人把一个全部被注释掉的测试文件从项目中删除。

实现一个优良的单元测试需要注意一下几点。

assert报错

使用assert来报错，而不要使用输出语句来输出信息判断。
一般在测试代码中经常会编写输出代码，然后通过人工观察来验证逻辑正确性。这样做平添了更多烦恼，当整个项目进行单元测试的时候，需要观察所有的输出是否与预期相同。
断言是单元测试中最基本的组成部分。当一个失败或者错误出现的时候，当前测试方法的执行流程会被终止，但是位于同一个测试类中的其他测试将会继续进行。

测试强度要大

对结果的验证要全面，不能有模糊的概念。
在断言中的判断条件要比较细致，逐条判断。尽可能的反应出函数的明确需求。

以最简单的运算类函数为例，一般能覆盖以下几种值就行了：

  1. 空值
  2. 普通合法值
  3. 边界合法值（最大/最小合法值）
  4. 边界非法值（最大/最小非法值）
  5. 普通非法值
  6. 极限值（最大/最小可能值）
  7. 特殊值（如果存在）
    列举Test Case本就是个头脑风暴的过程，只有对相应的业务和代码都有足够的了解，才能写出有意义的Test Case。

单元要小

测试的最大单元是类，一般是方法，只有单元小才能在出错时快速定位。
单元测试要保证最小单元的逻辑正确，不要与其他单元有耦合，单元之间的耦合在上一层去测试。

稳定

测试不能受到外界环境的影响。
