UNIX网络编程 读书笔记

发表于 2017-06-04 | 分类于 Note

UNIX网络编程 读书笔记

简介

首先回顾TCP/IP四层模型：由下到上依次为 网络接口层、网络层、传输层、应用层。

不同的协议层对数据包有不同的称谓，在传输层叫做段（segment），在网络层叫做数据报（datagram），在链路层叫做帧（frame）。数据封装成帧后发到传输介质上，到达目的主机后每层协议再剥掉相应的首部，最后将应用层数据交给应用程序处理。

POSIX可移植操作系统接口，Portable Operating System Interface，X标明其对Unix API的传承。

传输层：TCP、UDP和SCTP

用户数据报协议UDP

UDP不保证数据包会到达其最终目的地，不保证各个数据报的先后顺序跨网络后保持不变，也不保证每个数据报只到达一次。每个UDP数据报都有一个长度，而TCP是一个字节流协议，没有任何边界记录。

传输控制协议TCP

当TCP向另一端发送数据时，它要求对端返回一个确认。如果没有收到确认，TCP就自动重传数据并等待更长时间。在数次重传失败后，TCP才放弃。
TCP通过对其中每个字节关联一个序列号对所发送的数据进行排序。如果接收端TCP接收到来自对段的重复数据，它可以根据序列号判定数据是重复的，从而丢弃重复数据。
TCP提供流量控制（flow control）。TCP总是告知对端在任何时刻它一次能够从对端接受多少字节的数据，这称为通知窗口（advertised window）。在任何时刻，该窗口指出接受缓冲区中当前可用的容量，从而确保发送端发送的数据不会使接受缓冲区溢出。该窗口时刻动态变化。

TCP连接的建立和终止

三次握手：
1）客户端发送一个带SYN标志的TCP报文到服务器。（客户端connect阻塞）
2）服务器端回应客户端，同时带ACK标志和SYN标志。因此它表示对刚才客户端SYN报文的回应；同时又标志SYN给客户端，询问客户端是否准备好进行数据通讯。（服务端accept阻塞）
3）客户必须再次回应服务段一个ACK报文。（客户端connect返回，服务端接收到ack后accept返回）
(accept发生在三次握手在之后)

四次挥手：
1）客户端A发送一个FIN，用来关闭客户A到服务器B的数据传送。
2）服务器B收到这个FIN，它发回一个ACK，确认序号为收到的序号加1。和SYN一样，一个FIN将占用一个序号。
3）服务器B关闭与客户端A的连接，发送一个FIN给客户端A。
4）客户端A发回ACK报文确认，并将确认序号设置为收到序号加1。

状态

FIN_WAIT_1 表示客户端主动关闭，等待服务器确认。
FIN_WAIT_2 表示半关闭状态，等待服务器关闭，然后才最后关闭。
TIME_WAIT 表示收到了对方的FIN报文，并发送出了ACK报文，就等2MSL后即可回到CLOSED可用状态了。
CLOSE_WAIT 表示被动关闭，客户端关闭了，现在服务端要检查还有没有数据要发送。
LAST_ACK 表示客户端关闭了，服务端要发送完所有的数据。

为什么TIME_WAIT状态还需要等2MSL后才能返回到CLOSED状态？
一个数据包在网络中的最长生存时间为MSL。
虽然双方都同意关闭连接了，而且握手的4个报文也都协调和发送完毕，按理可以直接回到CLOSED状态（就好比从SYN_SEND状态到ESTABLISH状态那样）；但是因为我们必须要假想网络是不可靠的，你无法保证你最后发送的ACK报文会一定被对方收到，因此对方处于LAST_ACK状态下的SOCKET可能会因为超时未收到ACK报文，而重发FIN报文，所以这个TIME_WAIT状态的作用就是用来重发可能丢失的ACK报文。

流控制传输协议SCTP

SCTP在客户和服务器之间提供关联，并像TCP那样给应用提供可靠性、排序、流量控制以及全双工的数据传送。SCTP中使用“关联”一词取代“连接”是为了避免这样的内涵：一个链接只涉及两个IP地址之间的通信；一个关联代指两个系统之间的一次通信，她可能因为SCTP支持多宿而涉及不止两个地址。

端口号

TCP、UDP、SCTP这三种传输层协议通过16位整数的端口号来区分这些进程。
UNIX系统小于1024的端口被保留，只能被赋予特权用户进程。

一个TCP连接的套接字对(socket pair)是一个定义该链接的两个端点的四元组：本地IP地址、本地TCP端口号、外地IP地址、外地TCP端口号。标识每个端点的两个值成为一个套接字。

I/O复用：select和poll函数

TCP客户同时处理两个输入：标准输入和TCP套接字，遇到的问题是在客户阻塞于（标准输入上的）fgets调用期间，服务器进程会被杀死。服务器TCP虽然正确地给客户TCP发送了一个FIN，但是既然客户进程正阻塞于从标准输入读入的过程，它将看不到这个EOF，直到从套接字读时为止。这样的几次进程需要一种预先告知内核的能力，使内核一旦发现进程指定的一个或多个I/O条件就绪（也就是说输入已准备好被读取，或者描述符已能承接更多的输出），他就通知进程。这种能力称为I/O复用，是由select和poll这两个函数支持的。

I/O复用典型使用在下列网络应用场合：
-当客户处理多个描述符
-如果一个TCP服务器既要处理监听套接字，又要处理已连接套接字
-一个服务器既要处理TCP，又要处理UDP
-一个服务器要处理多个服务或者多个协议

I/O模型

5种I/O模型
阻塞式I/O模型、非阻塞式I/O模型、I/O复用、信号驱动式I/O、异步I/O

select

该函数允许进程指示内核等待多个事件中的任何一个发生，并只在有一个或多个事件发生或经历一段指定的时间后唤醒它。

  • 返回对应位仍然为1的fd的总数（只有那些可读、可写以及有异常条件待处理的fd位仍然为1。否则为0）
  • maxfdp1指定待测试的描述符个数，它的值是待测试的最大描述符加1。
  • readset、writeset、exceptset三个参数指定我们要让内核测试读、写和异常条件的描述符。（如果都为NULL可以用来实现一个定时器）
  • timeout告知内核等待所指定描述符中的任何一个就绪可花多长时间。有三种可能：
    1）永远等待下去，表示仅在有一个描述符准备好I/O时才返回；
    2）等待一段固定时间，表示在一段固定时间后有描述符准备好I/O就返回；
    3）根本不等待，表示有描述符就立即返回，又称为 轮询polling

一些结构体介绍：
fd_set表示一组文件描述符(fd)的集合，对于fd_set类型通过下面四个宏来操作：
FD_ZERO(fd_set fdset) 将指定的文件描述符清空
FD_SET(fd_set fdset) 用于在文件描述符集合中增加一个新的文件描述符
FD_CLR(fd_set fdset) 用于在文件描述符集合中删除一个文件描述符
FD_SET(fd_set fdset) 用于测试指定的文件描述符是否在该集合中

timeval结构用于描述一段时间长度，两个数据用于指定这段时间的秒数和微秒数，提高了分辨率

timeout == NULL表示情况1，永远等待下去，阻塞
timeout指向的结构体为非零时间表示等待一段固定时间
timeout指向结构体的值为0，表示根本不等待，只检测描述符是否有，有就返回，并不等待外部事件的发生

select使用

比如recv()，在没有数据到来调用它的时候，线程将被阻塞，如果数据一直不来，线程就要阻塞很久。这样显然不好。所以采用select来查看套节字是否可读，也就是是否有数据读了。

使用select函数的过程一般是:
先调用宏FD_ZERO将指定的fd_set清零，然后调用宏FD_SET将需要测试的fd加入fd_set，接着调用函数select测试fd_set中的所有fd，最后用宏FD_ISSET检查某个fd在函数select调用后，相应位是否仍然为1。

poll函数

poll的机制与select类似，与select在本质上没有多大差别，管理多个描述符也是进行轮询，根据描述符的状态进行处理，但是poll没有最大文件描述符数量的限制。poll和select同样存在一个缺点就是，包含大量文件描述符的数组被整体复制于用户态和内核的地址空间之间，而不论这些文件描述符是否就绪，它的开销随着文件描述符数量的增加而线性增大。

  • fdarray表示被监视的多个文件描述符，pollfd结构体指定了一个被监视的文件描述符
  • nfds表示数组fdarray元素的个数
  • timeout指定等待的毫秒数。

每一个pollfd结构体指定了一个被监视的文件描述符，结构体的events域是监视该文件描述符的事件掩码，由用户来设置这个域。revents域是文件描述符的操作结果事件掩码，内核在调用返回时设置这个域。

poll的实现和select非常相似，只是描述fd集合的方式不同，poll使用pollfd结构而不是select的fd_set结构，其他的都差不多。
