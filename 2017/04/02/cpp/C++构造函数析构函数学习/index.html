C++构造函数析构函数学习

Posted on 2017-04-02 | In Blog

深入理解构造函数、析构函数的工作

this指针

C++中类的this指针就是一个指向类的指针。定义了一个class A;，那么this的类型就是A*，对this解引用就是一个A类型的变量。

this指针的位置和作用域

通过对程序反汇编，可以看到this指针会被放到eax寄存器中(不同编译器可能会采取不同处理方式，在VS中会放到eax寄存器中）。他指向的是连续存放的类的第一个成员变量（如果类中有虚函数的话，虚表指针在类对象内存空间第一个，所以this指针会指向虚表指针）。所以一个对象的this指针并不是对象本身的一部分，不会影响sizeof的结果。

this指针在对象被创建之后才被赋值，所以在static成员函数中不会有this指针。
this的作用域是在类的内部，当类的非静态成员函数访问非静态成员变量时，编译器会在编译时自动将对象本身的地址作为一个隐含参数传递给类的非静态成员函数。
实际上成员函数默认第一个参数为A* const register this

同时this指针是一个const指针，不能对this指针进行赋值、加减操作。

this指针的使用

  1. 在类的非静态成员函数中返回类对象本身return *this;
  2. 当参数名与成员变量名相同时，作为区别防止作用域覆盖掉成员变量this.data = data;

构造函数

构造函数有没有返回值

结论是构造函数没有返回值。
我们知道对于一个函数来说，实际上都会有一个返回值，即使没有返回值也要写上void。
但是构造函数的函数声明中是没有返回值这个模块的。
通过对C++代码反汇编，可以看出实际上构造函数调用的时候返回了一个值，但是这个值被保存到了eax寄存器中。根据之前对this指针的研究，this指针在eax寄存器中，不难猜到构造函数实际上返回了类的this指针。但是类的this指针是存放在寄存器中，其作用域也只在类的内部，对其取址操作会报错。所以不能在实际编程中外部使用这个this指针。

构造函数抛出异常

由于构造函数没有返回值，我们不能判断是否成功构造对象。只能通过抛出异常来判断（唯一方法）
构造函数抛出异常将导致对象的析构函数不能执行。
抛出异常的位置随机，当发生部分构造的时候必须保证前面申请的资源被按析构顺序释放掉。

new

在面试中经常会问道new干了什么，对于这个问题的可以这么回答：
new内含两阶段操作：调用::operator new配置内存；调用构造函数构造对象。
同理，对于delete来说也有两阶段操作：调用析构函数析构对象；调用::operator delete释放内存。

operator new

一般我们使用new是作为操作符，使用new在堆上创建一个对象。它实际上做了三件事情：

  1. 获取一块内存空间
  2. 调用构造函数
  3. 返回正确的指针
    如果我们在堆上创建的是一个内置类型的变量，那么第二步被省略。

作为操作符，是可以被重载的，在C++中不能被重载的操作符只有六个(. .* ?: :: sizeof typeid)

一般重载了new也会一起重载delete，保证资源被正确获取释放。

总结：operator new默认行为是请求分配内存，如果成功则返回此内存地址，如果失败则调用一个new_handler，然后再重新请求分配内存。

定位构造

上面那种重载new的方式只是简单的在默认的new操作符上外包了一层，不属于真正意义上的重载。
在C++中，可以通过定位构造(placement new)来实现三步操作中的第二步重载

代码new(pa) A(3); placement new 实现了在指定内存地址上用指定类型的构造函数来构造一个对象的功能，后面A(3)就是对构造函数的显式调用。这块指定的地址既可以是栈，又可以是堆，placement new对此不加区分。如果在栈上调用，系统会自动释放对象，如果在堆上调用，就要显式调用析构函数pa->~A();

placement new的作用就是：创建对象(调用该类的构造函数)但是不分配内存，而是在已有的内存块上面创建对象。用于需要反复创建并删除的对象上，可以降低分配释放内存的性能消耗。

定义一个只能在堆上（栈上）生成对象的类

类对象的创建方法有两种：静态创建A a(10);和动态创建A* pa = new A(10);

  1. 只能在堆上生成对象
    那就表示只能通过new操作符来动态创建对象。不能在外部直接调用类的构造函数。
    先考虑将构造函数设为private，外部就无法调用。但是我们知道new操作符实际上分为三步：获取一块内存空间、调用构造函数、返回正确的指针。在new操作符中还是要调用构造函数，如果将构造函数设为private，连动态创建都不行了。

换一个思路：当对象生命周期结束，如果是在栈上创建的对象，系统自动调用析构函数释放对象内存，而对于堆来说，要手动delete。所以如果我们将析构函数设置为private，编译器无法从外部自动调用析构函数来释放内存。在编译器为类对象分配栈空间的时候，先要检查类的析构函数的访问性，如果类的析构函数时私有的，编译器不会在栈上为类对象分配内存。同时，delete也会自动调用析构函数，我们也不能使用delete了，为了能够手动释放内存，我们要编写一个新函数，来替代delete，手动释放内存

最后，为了能够继承，将析构函数设为protected的。

还可以换种方式：将构造函数设置为private，然后编写两个静态函数分配和释放类对象

  1. 只能在栈上生成对象
    不能使用new运算符，将其设为private即可

捕获构造函数成员初始化列表产生的异常

为了处理构造函数成员初始化列表产生的异常，必须将构造函数编写为函数测试块（function try block）。

关键字try出现在成员初始化列表之前，测试块的复合语句包围了构造函数的函数体。catch子句既可以处理从成员初始化列表中抛出的异常，也可以处理从构造函数函数体中抛出的异常。

析构函数

析构函数抛出异常

对于析构函数来说，是可以抛出异常的，但是不建议这么做。
程序可能因为发生异常而终止，这个时候需要自动销毁对象而调用析构函数，同时这个时候析构函数又抛出异常，会产生死循环。
当析构函数一定要抛出异常的时候，要确保问题都被析构函数catch消灭掉。

虚析构函数

虚函数在C++中广泛的使用，但是如果有人问虚函数有什么缺点？怎么回答呢
首先回答不同的虚函数实现在不同的类里，在内存里较分散，导致调用时cache命中率下降，影响性能

我们一般为了继承会使用虚函数，唯一的缺点就是多了一个虚表指针，增加了类对象的大小，但是为了继承这样做是必要的。所以这个缺点不存在。
但是经常我们会使用虚析构函数，并且很多类没有继承关系都会编写虚析构函数，这样做就不合适了。STL容器中都没有虚析构函数，不过为了设计抽象类定义纯虚析构函数也是合理的。
不要无故的声明虚析构函数

构造函数中调析构函数

构造函数和析构函数都能够显式调用

如果类中没有资源，不需要在析构函数中释放，那么调用析构函数多次都可以，不会出现程序上的问题，类对象会在生命周期结束自动被系统释放。如果有资源会出现重复删除，会报错。所有在构造函数中调用析构函数，如果类中没有什么资源，不出现多次删除，实际上应该是没有大影响的。
