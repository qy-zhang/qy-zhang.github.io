C++头文件和源文件

发表于 2017-04-25 | 分类于 Blog

我们知道一般声明写在.h文件，定义实现写在.cpp文件，C++为了实现代码分离，设计了头文件和源文件。总结一下头文件、源文件里面该写什么。

几个基本概念

声明和定义

上面我们提到了 声明写在.h文件，定义实现写在.cpp文件

声明是向编译器介绍名字——标识符。它告诉编译器“这个函数/变量在某个地方（具体不知道）可以找到，它的模样是什么样子”
定义是要编译器在定义的地方为变量/函数分配存储空间。

在C++中，在一个作用域中重复一个声明是合法的。可以多次声明，但不能多次定义。

内部链接和外部链接

内部链接：如果一个名称对于它的编译单元（一个cpp文件）来说是局部的，并且在连接的时候不可能与其它编译单元中的同样的名称相冲突，则这个名称具有内部链接。即具有内部连接的名称不会被带到目标文件中。
外部链接：在一个多文件程序中，如果一个名称在连接时可以和其他编译单元交互，那么这个名称就具有外部链接。即具有外部连接的名称会引入到目标文件中，由连接程序进行处理。这种符号在整个程序中必须是惟一的。

我们知道C++中声明和定义是可以分开的，可以一个函数声明定义放在b.cpp中，在a.cpp只需再声明一下这个函数，就可以在a.cpp中使用这个函数了：

而通过之前的了解，我们知道每个编译单元间是相互独立不知道彼此的存在的，那么a.cpp又是如何知道show函数的定义的呢？
其实在编译一个编译单元(.cpp)生成相应的obj文件过程中，编译器会将分析这个编译单元(.cpp)，将其所能提供给其他编译单元(.cpp)使用的函数，变量定义记录下来。而将自己缺少的函数，变量的定义也记录下来。所以可以认为a.obj和b.obj记录了以下的信息：
a.obj : 我能提供main函数的定义、我需要show函数的定义
b.obj : 我能提供show函数的定义
然后在链接器连接的时候就会知道a.obj需要show函数定义，而b.obj中恰好提供了show函数的定义，通过链接，在最终的可执行文件中我们能看到show函数的运行

分类

具有内部链接的定义包括:

  • 所有的声明
  • 加static前缀的全局变量定义。static int x;，static限制作用域在文件内部。
  • 名字空间中const常量定义。
  • 枚举类型的定义。enum Boolean {NO,YES };
  • 类的定义。class Point {...};
  • 内联函数的定义。inline int operator==(){...}
  • union的定义。
    具有外部链接的定义包括:
  • 类的非内联成员函数。（不在类中直接实现的函数）
  • 类的static成员变量，位于全局区所有对象共享。
  • 非内联，非静态的自由函数。
  • 非静态的全局定义。

解释几点：

  1. 声明式内部链接
    声明本身不会将任何符合引入目标文件，所以声明都是内部连接的。
    但是可以使用extern关键字声明来实现外部链接

  2. const常量是内部链接
    C++规定，有const修饰的变量，不但不可修改，还都将具有内部链接属性，也就是只在本文件可见。（这是原来C语言的static修饰字的功能，现在const也有这个功能了。）
    又补充规定，extern const联合修饰时，extern将压制const这个内部链接属性。
    于是，extern char const s[];将仍然有外部链接属性，但是还是不可修改的。 具体使用方法如下:
    在cpp文件中:extern const int a=1;
    在要使用该常量的cpp文件中:extern const int a;//也可以将其放到头文件中
    当用extern修饰const变量将强制为其分配内存，因为extern表示采用外部链接，因此其必须有某个地址保存其值。

  3. 类的定义是内部链接
    如果不是，想象一下你在4个cpp文件中include定义了类Base的头文件，在4个编译单元中的类Base都有外部链接，在连接的时候就会出错。

  4. 类的非内联成员函数是外部链接
    实现放在cpp文件中，肯定是外部链接

头文件的一些特别点

通过上面我们知道了头文件中是放一些外部链接的函数/变量

防止重复包含

头文件中不止会放声明，也会放一些定义，只要是外部链接的就行，但是由于不知道有多少程序需要使用头文件，每次引用头文件都会定义一份这些外部链接的函数/变量，所以要防止重复定义，就要防止重复包含。

include作用

include预处理器命令告诉预处理器打开指定的头文件并在#include语句所在的地方插入头文件。

什么是编译单元呢？简单来说一个cpp文件就是一个编译单元。
在集成式的IDE中，我们往往点击一下运行便可以了，编译的所有工作都交给了IDE去处理，往往忽略了其中的内部流程。

事实上编译每个编译单元(.cpp)时是相互独立的，即每个cpp文件之间是不知道对方的存在的（不考虑#include “xxx.cpp” 这种奇葩的写法），编译器会分别将每个编译单元(.cpp)进行编译，生成相应的obj文件，然后链接器会将所有的obj文件进行链接，生成最终可执行文件

模板类型

对于类模板，在定义它的时候，编译器并不会对它进行编译，因为它没有一个实体可用。
只有类模板被具体化（specialization）之后（用在特定的类型上），编译器才会根据具体的类型对模板进行编译。
因为模板的这种特殊性，它并没有自己的准确定义，因此我们不能把它放在.cpp文件中，而要把他们全部放在.h文件中进行书写。这也是为了在模板具体化的时候，能够让编译器可以找到模板的所有定义在哪里，以便真正的定义方法。

Self-contained 头文件

头文件应该能够自给自足（self-contained，也就是可以作为第一个头文件被引入），以 .h 结尾。至于用来插入文本的文件，说到底它们并不是头文件，所以应以 .inc 结尾。不允许分离出 -inl.h 头文件的做法。

所以我们可以看出有些只是作为文本插入到代码的某个地方的情况，就不要写成.h结尾，而写成.inc结尾
如果 .h 文件声明了一个模板或内联函数，同时也在该文件加以定义。如果某函数模板为所有相关模板参数显式实例化，或本身就是某类的一个私有成员，那么它就只能定义在实例化该模板的 .cc 文件里。

define保护

所有头文件都应该使用 #define 来防止头文件被多重包含, 命名格式当是: _H_

为保证唯一性，头文件的命名应该基于所在项目源代码树的全路径。例如，项目foo中的头文件 foo/src/bar/baz.h ，对应命名格式是FOO_BAR_BAZ_H_，FOO为项目名，BAR为文件路径，BAZ为文件名。可按如下方式保护：

前置声明

所谓前置声明（forward declaration）是类、函数和模板的纯粹声明，没伴随着其定义。

使用前置声明的优点：
前置声明能够节省编译时间，多余的 #include 会迫使编译器展开更多的文件，处理更多的输入。
前置声明能够节省不必要的重新编译的时间。 #include 使代码因为头文件中无关的改动而被重新编译多次。
