Nginx模块开发与结构解析 读书笔记

Posted on 2017-05-15 | In Note

Nginx（发音同 engine x）是一款轻量级的Web 服务器/反向代理服务器及电子邮件（IMAP/POP3）代理服务器，并在一个BSD-like 协议下发行。其特点是占有内存少，并发能力强。

nginx平台初探

初探nginx架构

nginx在启动后，在unix系统中会以daemon的方式在后台运行，后台进程包含一个master进程和多个worker进程。
注：daemon表示Linux下的守护进程，守护进程就是一直在后台运行的进程，让他像服务一样成为后台任务。
Linux命令行下通过在命令尾部加上符号&，启动的进程就会变成为“后台任务”，如果要让正在运行的”前台任务”变为”后台任务”，可以先按ctrl + z，然后执行bg命令。
参考博客

守护进程的创建：创建子进程，然后父进程先退出，子进程就变为孤儿进程，并由init进程作为其父进程收养。
孤儿进程和僵尸进程的区别：僵尸进程是指子进程先于父进程结束，而且父进程没有函数调用wait()或waitpid()等待子进程结束，使得子进程的进程列表信息没法回收。

master进程主要用来管理worker进程，包含：接收来自外界的信号，向各worker进程发送信号，监控worker进程的运行状态，当worker进程退出后(异常情况下)，会自动重新启动新的worker进程。而基本的网络事件，则是放在worker进程中来处理了。多个worker进程之间是对等的，他们同等竞争来自客户端的请求，各进程互相之间是独立的。一个请求，只可能在一个worker进程中处理，一个worker进程，不可能处理其它进程的请求。nginx的进程模型，可以由下图来表示：

每个worker进程都是从master进程fork过来，在master进程里面，先建立好需要listen的socket（listenfd）之后，然后再fork出多个worker进程。所有worker进程的listenfd会在新连接到来时变得可读，为保证只有一个进程处理该连接，所有worker进程在注册listenfd读事件前抢accept_mutex，抢到互斥锁的那个进程注册listenfd读事件，在读事件里调用accept接受该连接。当一个worker进程在accept这个连接之后，就开始读取请求，解析请求，处理请求，产生数据后，再返回给客户端，最后才断开连接，这样一个完整的请求就是这样的了。我们可以看到，一个请求，完全由worker进程来处理，而且只在一个worker进程中处理。

深入Nginx

Nginx基础架构

事件驱动架构

所谓事件驱动架构，就是由一些事件发生源来产生事件，由一个或者多个事件收集器来收集、分发事件，然后许多事件处理器会组册自己感兴趣的事件，同时会“消费”事件。
对于传统的Web服务器来说，事件驱动仅限于TCP连接建立、关闭事件上，一个连接建立以后，在其关闭之前的所有操作都不再是事件驱动，这时会退化成按序执行每个操作的批处理模式。这种方式可以把一个进程或线程作为事件消费者，当一个请求产生事件被该进程处理时，知道这个请求处理结束时进程资源都将被这一个请求所占用。

Nginx不会使用进程或线程作为时间消费者。只有事件收集、分发器才有资格占用进程资源，他们会在分发某个事件时调用事件消费模块使用当前占用的进程资源。

传统Web服务器与Nginx间的重要区别：
前者是每个事件消费者独占一个进程资源，后者的事件消费者只是被事件分发者进程短期调用而已。
Nginx这种设计能够使得服务器的网络吞吐量增大，但是每个事件消费者不能有阻塞行为，否则将会由于长时间占用事件分发者进程而导致其他事件得不到及时响应。

请求的多阶段异步处理

只有把请求分为多个阶段，才有所谓的异步处理。
把一个请求的处理过程按照事件的触发方式划分为多个阶段，每个阶段都可以由事件收集、分发器来触发。划分方式一般是找到请求处理流程中的阻塞方法。

管理进程、多工作进程设计

Nginx采用一个master管理进程、多个worker工作进程的设计方式。

这样设计带来以下优点：

  • 利用多核系统的并发处理能力：多个worker进程平等，并发处理
  • 负载均衡：一个请求到来时更容易被分配到负载较轻的worker工作进程中处理。
  • 管理进程会负责监控工作进程的状态，并负责管理其行为：管理进程是用来启动、停止、监控或停止其他行为来控制工作进程，这样提高了系统的可靠性。

事件模块

epoll事件驱动模块

设想一个场景：有100万用户同时与一个进程保持着TCP连接，而每一时刻只有几十个TCP连接是活跃的（接收到TCP包），也就是说每一时刻进程只需要处理这100万连接中的一小部分连接。那么如何才能高效地处理这种场景呢？
进程每次询问操作系统收集有事件发生的TCP连接时，把这100万个连接告诉操作系统，然后由操作系统找出其中有事件发生的几十个连接。select和poll事件驱动就是这样做的。
我们可以发现这样做效率特别低：每次收集事件时，都要把这100万连接的套接字传给操作系统（涉及到用户态到内核态的大量复制）。
epoll把原先的select和poll分成了三个部分：调用epoll_create创建一个epoll对象、调用epoll_ctl想epoll对象注册这100万个连接的套接字、调用epoll_wait收集发生事件的连接。epoll_wait没有100万个连接，内核不需要遍历所有的连接了。

Linux内核实现：

每个epoll对象都有一个独立的eventpoll结构体，这个结构体会在内核空间中创造独立的内存，用于存储使用epoll_ctl方法向epoll对象中添加进来的事件。每个事件都会挂到rbr红黑树上，重复添加的事件会跳过（红黑树性质）
当调用epoll_wait检查是否有发生事件的连接时，只是检查eventpoll对象中的rdllist双向链表是否有epitem元素而已，如果rdllist链表不为空，就把这里面的事件赋值到用户态。因此epoll_wait效率非常高。

进程间的通信机制

Nginx由一个master进程和多个worker进程组成，但master进程或者worker进程中并不会再创建线程（Nginx的多线程机制一直停留在测试状态）
Nginx框架使用了3种消息传递方式：共享内存、套接字、信号；在多个进程访问共享资源时，使用了3中同步方式：原子操作、信号量、文件锁。

共享内存

Nginx定义了ngx_shm_t结构体用于描述一块共享内存：

先以mmap为例说明Linux是怎样向应用程序提供共享内存的：
void *mmap(void *start, size_t length, int prot, int flags, int fd, off_t offset);
mmap可以将磁盘文件映射到内存中，直接操作内存时Linux内核将负责同步内存和磁盘文件中的数据
fd参数就指向需要同步的磁盘文件；offset代表从文件的这个偏移量出开始共享；length表示要在内存中开辟的地址空间大小；prot参数表示操作共享内存的方式（只读或者可读可写）；start参数说明共享内存起始映射地址。

在nginx中使用ngx_shm_alloc来申请共享内存，实现就是调用mmap函数。

原子操作

能够执行原子操作的原子变量只有整形，包括无符号整形和有符号整形，这两种类型都使用了volatile关键字告诉C编译器不要做优化。

基于原子操作，Nginx实现了一个自旋锁。自旋锁是一种非睡眠锁：在拿不到锁时，这个进程的代码会一直在自旋锁代码处执行，直到其他进程释放了锁且当前进程获取到锁后，代码才继续向下执行。
自旋锁要解决的问题时进程使用锁的时间非常短（如果使用时间长，一直执行会占用大量CPU资源）。Nginx的worker进程最好不要进入睡眠状态，因为它非常繁忙，在这个进程的epoll上可能会有几百万的TCP连接等待着处理，进程一旦睡眠后必须等待其他事件的唤醒，这中间极其频繁的进程间切换带来的负载消耗可能让用户无法接受。

ngx_spinlock函数实现了自旋锁

Nginx频道

ngx_channel_t频道是Nginx中master进程和worker进程之间通信的主要工具，使用本机套接字实现。
ngx_channel_t结构体格式：

使用socketpair方法创建父子进程间使用的套机字。这个方法可以创建一对关联的套接字。
master通过socketpair产生的套接字发送命令，来启动、停止worker子进程。
ngx_write_channel发送消息，ngx_read_channel接收消息。
worker使用ngx_add_channel_event方法把接收频道消息的套接字添加到epoll中，当接收到父进程消息时子进程会通过epoll的事件回调相应的handler方法来处理这个频道消息。

信号量

信号量是用来保证两个或多个代码段不被并发访问，是一种保证共享资源有序访问的工具。使用信号量作为互斥锁有可能导致进程睡眠，因此要谨慎使用。

最初的信号量sem值为0，调用sem_post方法会把sem值加1，这个操作不会有任何阻塞；调用sem_wait方法会把信号量sem的值减1，如果sem值已经小于或等于0了，则阻塞住当前进程（进程会进入睡眠状态），知道其他进程将信号量sem的值改变为正数后，这是才能通过将sem减1而使得当前进程继续向下执行。因此sem_post方法可以实现解锁的功能，而sem_wait可以实现加锁的功能。

互斥锁


ngx_shmtx_lock会阻塞进程，ngx_shmtx_trylock不会阻塞进程。
他们都通过操作ngx_shmtx_t类型的结构体来实现互斥操作。

结构体对应着三种不同的实现：

  1. 支持原子操作，但是不支持信号量
  2. 支持原子操作，同时支持信号量
  3. 不支持原子操作，使用文件锁实现
