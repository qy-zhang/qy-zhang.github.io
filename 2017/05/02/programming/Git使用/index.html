Git使用

Posted on 2017-05-02 | In Blog

恰当的使用git，对团队开发和项目管理起着至关重要的作用。
首先团队中使用Git同步要正确地使用分支、其次每次提交git commit的时候要有规范的注释

Git分支

每次开发新功能，都应该新建一个单独的分支

主分支master

代码库应该有且仅有一个主分支master。所有提供给用户使用的正式版本，都在这个主分支上发布。
Git主分支的名字，默认叫做master。它是自动建立的，版本库初始化以后，默认就是在主分支在进行开发。

开发分支develop

主分支只用来分布重大版本，日常开发应该在另一条分支上完成。
每次新开始一个项目，就创建一个开发分支，之后的开发工作就在develop分支上完成。

如果开发进行到一定的进度想要发布一个版本，就在master分支上对develop分支进行合并操作。

默认情况下，Git执行”快进式合并”（fast-farward merge），会直接将master分支指向develop分支。使用–no-ff参数后执行合并会生成一个新的节点。

临时分支

临时性分支用于应对一些特定目的的版本开发。从develop分支中分出来，开发完合并到develop中。最后这些临时分支会被删除。

临时分支主要有以下几种

  1. 功能分支feature：发布一个新功能
  2. 修补bug分支fixbug：修正bug

Git注释

每次推送要正确地编写注释信息。

commit规范

提交commit时要注意规范。
不要使用git commit -m update这种无用注释信息。-m参数编写注释不能换行。
可以直接使用git commit然后在新打开的窗口中编写注释信息，最好加上–verbose参数，可以看到修改的信息

良好的commit可以参考以下几点

  1. 第一行不超过50个单词，用陈述句
  2. 第二行空一行，这样在浏览代码的时候看到的注释信息是第一行
  3. 后面罗列出改动原因、主要变动、以及需要注意的问题
  4. 最好添加上修复bug的查阅资料（比如浏览了stackoverflow看到了解决bug思路）

Git推送拉取

Git中最常用的操作应该就是push pull了

新建一个仓库

一般首先从远程主机克隆一个版本库
git clone <website>
或者是初始化当前文件夹，然后添加一个远程仓库（repository）

拉取

远程主机的版本库有了更新commit，需要将这些更新取回本地
git fetch origin master

而git pull是取回远程主机某个分支的更新，再与本地的指定分支合并。
一般认为pull = fetch + merge
尽量使用git fetch和git merge来替代git pull

以上命令相当于git pull origin master，如果加上了-rebase操作，相当于是使用git rebase来代替git merge

merge 是合并的意思，rebase是复位基底的意思。merge操作会生成一个新的节点，之前的提交分开显示。而rebase操作不会生成新的节点，是将两个分支融合成一个线性的提交。

实际中使用

脚本

可以在git目录下新建一个脚本 push.sh ，每次想要提交的时候运行脚本即可，这么做的缺陷是git commit信息不完善，便于简单提交，以后再通过git rebase命令合并commit，完善commit信息。

git add添加不了

有时候git status会发现modified: xxx(modified content)这种内容，通过git add添加不上去
其实是这个目录没有被跟踪，目录下有一个.git目录。解决方法是删除这个.git目录

修改远程仓库地址

有三种方法：

  1. 修改：git remote set-url origin [url]
  2. 先删后加：git remote rm origin之后git remote add origin [url]
  3. 直接修改config文件：修改/.git/config文件中的[remote "origin"]url信息即可
