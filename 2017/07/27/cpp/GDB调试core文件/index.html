GDB调试core文件

Posted on 2017-07-27 | In Note

在程序运行的过程中，有的时候我们会遇到segment fault(段错误)这样的错误。这种解决起来比较困难，因为由于段错误，程序直接退出了，没有任何的栈、trace信息输出。在Linux中core dump专门来解决这一类问题。

core dump介绍

core dump又叫核心转储，当程序运行过程中发生异常退出时，由操作系统把程序当前的内存状况存储在一个core文件中，这一个过程就叫做叫core dump。

core文件

当一个程序崩溃时，在进程当前工作目录的core文件中复制了该进程的存储图像。core文件仅仅是一个内存映象（同时加上调试信息），主要是用来调试的。

当程序接收到一些UNIX信号时，就会终止程序，并且产生core文件，常见的信号主要是如下几种：

信号名称 条件
SIGABRT 进程异常终止
SIGFPE 此信号表示一个算术运算异常，例如除以0，浮点溢出等
SIGSEGV 指示进程进行了一次无效的存储访问。SEGV表示段违例segmentation violation

当程序崩溃时，core文件在进程当前工作目录下创建，与可执行程序在相同路径下。一般是以core开头，后面会跟上进程号（不同的系统实现不同）

造成segment fault，产生core dump的可能原因

  1. 内存访问越界
    主要有以下几种可能：数组访问越界；字符串没有使用结束符，访问越界；使用了不安全的c库函数strcpy等，导致dst字符串越界
  2. 多线程使用了线程不安全的函数
  3. 非法指针
    使用空指针
  4. 栈溢出
    局部变量过大，递归过深

Linux下配置ulimit

默认状态下系统是不会产生core文件的，我们通过ulimit命令来进行设置。

ulimit命令用来限制系统用户对shell资源的访问。支持以下各种类型的限制：所创建的内核文件的大小、进程数据块的大小、Shell 进程创建文件的大小、内存锁住的大小、常驻内存集的大小、打开文件描述符的数量、分配堆栈的最大大小、CPU 时间、单个用户的最大线程数、Shell 进程所能使用的最大虚拟内存。同时，它支持硬资源和软资源的限制。

假设有这样一种情况，当一台 Linux 主机上同时登陆了 10 个人，在系统资源无限制的情况下，这 10 个用户同时打开了 500 个文档，而假设每个文档的大小有 10M，这时系统的内存资源就会受到巨大的挑战。
所以通过限制资源的访问来确保系统正常运行。

ulimit -a 显示当前资源限制的设定
ulimit -c <core文件上限> 设置core文件的最大值

产生core文件

编写一个程序，修改不属于该程序的地址内容，这样会触发段错误segment fault

编译文件gcc -g -Wall test.c -o test可以正常编译，会有一个警告而已。
-Wall生成所有警告信息；-g产生调试信息；-o指定输出文件的文件名

运行程序./test程序会退出，并且控制台显示Segment Fault(core dump)，就说明触发了core dump了。
在当前目录下会有一个core文件，可以通过gdb调试了。
