private方法导致CGLib空指针

Posted on 2019-02-01 | In Note

问题描述

commonapi项目中TestController中访问private方法会抛空指针，public方法就正常

结论

TestController被cglib动态代理，cglib实现是继承被代理类，所以cglib代理生成的对象中不会有private方法

而@RequestMapping是可以将访问的url映射到private方法上的，当访问public方法时，spring容器中的通过cglib动态代理生成的TestController对象中可以继承public方法，所以可以正常访问

当访问private方法时，在cglib动态代理生成的TestController对象中没有这个方法，会使用原生的未被代理的对象，此时这个对象里的属性 redisDealCacheClient 等都没有被spring容器管理，所以是null

印证

实际到代码中确认：

  1. 断点打到 AbstractClassGenerator.generate(AbstractClassGenerator.ClassLoaderData data) 方法，找到字节数组

主要关注 byte[] b = this.strategy.generate(this); 这一句，断点打到这儿

  1. 在服务启动的时候打断点，spring容器在启动的时候需要实例化bean，断点设置当遇到TestController这个bean时停止

  2. 将字节数组转换输出

  3. 然后将encode出来的value值拷贝出来，再通过程序解码

  4. 最后将cc.class放到IntelliJ中即可查看通过Cglib动态代理生成的类对象所包含的内容了。
    里面是没有private方法的

几点疑问

  1. 为什么controller类会被cglib代理
  2. aggregation代码中同样在TestController写private方法，可以正常访问，不报空指针

正常情况下Controller类被Spring容器初始化之后，是不会被cglib动态代理的，所以写private方法一样可以正常访问，不会有空指针问题
一定是commonapi项目中使用了什么组件，使得所有的Controller都被cglib动态代理，而aggregation项目中没有使用到

通过查看commonapi中的调用堆栈，确定是ControllerLogCollector这个切面进行了处理，这个切面是mdp-boot-starter-log组件中的。

为了印证这个想法，在c-aggregation项目中也配置上mdp-boot-starter-log
首先同commonapi一样，在c-aggregation的 spring-servlet.xml 中加上了下面两个bean，然后添加了文件mdplog.properties

查看bean的类HttpAutoConfiguration ，发现在这个bean中实例化了ControllerLogCollector

发现还是没有被cglib代理，将上面两个bean的配置放到父servlet applicationContext.xml 中再次启动项目

问题复现，private方法中注入的对象为null
